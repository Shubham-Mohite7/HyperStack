# HyperStack

AI-powered tech stack recommendation engine. Describe your project in plain English and receive a confidence-scored, layer-by-layer recommendation generated by Llama 3.3 70B via Groq.

## Local development setup

### 1. Clone and install

```bash
git clone https://github.com/yourname/hyperstack.git
cd hyperstack
npm install
```

### 2. Configure environment variables

```bash
cp apps/api/.env apps/api/.env.local
```

Open `apps/api/.env` and set:

```
GROQ_API_KEY=your_key_here
GROQ_MODEL=llama3-70b-8192
PORT=3001
CLIENT_ORIGIN=http://localhost:5173
```

Get a free Groq API key at https://console.groq.com

### 3. Run both apps

```bash
npm run dev
```

This starts:
- Frontend at http://localhost:5173
- API server at http://localhost:3001

The Vite dev server proxies all `/api` requests to the Express server,
so you never need to configure cross-origin URLs in development.

## Deploying to Vercel

### Quick Deploy (Recommended)

```bash
# Use the deployment script
./deploy.sh
```

### Manual Deploy

```bash
npm install -g vercel
vercel --prod
```

### Environment Variables

In the Vercel dashboard, add these environment variables:

| Variable         | Value                        |
|------------------|------------------------------|
| GROQ_API_KEY     | Your Groq API key            |
| GROQ_MODEL       | llama-3.3-70b-versatile      |
| CLIENT_ORIGIN    | https://your-project.vercel.app |

### Deployment Files

- `vercel.json` - Handles routing and serverless configuration
- `DEPLOYMENT.md` - Detailed deployment guide
- `deploy.sh` - Automated deployment script

The `vercel.json` at the root handles routing: all `/api/*` requests
are sent to the Express handler, all other paths serve the React app.

## Extending the knowledge base

Add entries to `apps/api/lib/knowledgeBase.js`. Each entry follows this shape:

```js
{
  id:             "unique_id",
  name:           "Display name",
  layer:          "Frontend | Backend | Database | DevOps | Auth | AI/ML | Queue | Real-time | Monitoring",
  description:    "One-sentence summary",
  best_for:       ["use case 1", "use case 2"],
  avoid_when:     ["anti-pattern 1"],
  scale_ceiling:  "startup | large | massive | enterprise | unlimited",
  learning_curve: "low | medium | high",
  cost:           "free | free tier | pay per use | paid",
  maturity:       "experimental | beta | production",
  tags:           ["keyword1", "keyword2"],
}
```

Tags drive retrieval relevance вЂ” the more precise the tags, the better
the system matches them to project descriptions.

## Architecture decisions

**Why keyword scoring instead of vector embeddings?**

The API runs as a Vercel serverless function with a cold start budget under
500ms. Loading a sentence-transformer model would exceed this. The keyword
TF-IDF approach is fast, transparent, and produces accurate results when
combined with the LLM's contextual scoring in Stage 2 of the pipeline.

**Why three separate LLM calls instead of one?**

Splitting extraction, scoring, and report generation into separate prompts
with distinct system roles significantly improves output quality. A single
monolithic prompt asking the LLM to do all three tasks produces lower-fidelity
requirements extraction and less precise scoring. The added latency (~3-5s)
is acceptable given the value of the output.

**Why sessionStorage for result persistence?**

React Router state is lost on hard refresh. sessionStorage is a lightweight
solution that avoids introducing a state management library. Results are
scoped to the browser tab and cleared automatically when the session ends.
